Input Stage
Step 01 · Source Input
Centralised Email Box & File Path Intake
All source billing files are routed to a centralised email inbox. SPOC no longer monitors the inbox — Zai takes over from here. Files also arrive via a dedicated SAP file path for applicable clients.
Centralised Email Box
Dedicated SAP File Path
Note: Some transactions and documents are processed via email; others are submitted directly via SAP file path, depending on the client configuration.
Randstad
Zai picks up from centralised inbox / file path
Zai Intelligence Layer
Step 02 · Zai · Data Extraction
Zai Reads Files & Extracts Key Values
Zai reads the email and attachments (or ingests from the dedicated file path) and attempts to extract the following key identifiers from the source document:
Deputee ID
Client Name
Contract Code
Client Code
EW ID
Talent ID
Any one or more of these key values may be present in the document. Zai scans the full content to identify whichever fields are available.
Zvolv · Zai
Step 03 · Zai · Check — Key Value Identification
Can Zai Identify Key Values? → If No: Escalate to SPOC
If Zai cannot extract any of the required key values from the source file, the transaction is escalated immediately.
Exception 1: If no key value data can be fetched from the source file, the transaction is pushed to the Invoice SPOC for manual handling. This is tracked and audited — volume of such cases and reasons are logged.
Zvolv · Zai
Step 04 · Zai · Client Rule Engine
Client Rule Engine Lookup — Identify Billing Process Type
Using the extracted key values, Zai queries the Client Master (maintained in Zvolv, synced from RD system) to identify the client and determine the applicable billing process type.
The Client Master holds each client's Billing Transaction Type, Active status, and Billing Cycle — Zvolv maintains this as the single source of truth for all client billing rules.
Zvolv · Zai
Step 05 · Zai · Check — Client Master Validation
Client Found in Master? Active? → If No / Inactive: Escalate
Two checks are performed against the Client Master before proceeding to template generation.
Exception 2: If the client is not found in the Client Master, the transaction is routed to Invoice SPOC with the reason logged for audit.
Exception 3: If the client is marked as Inactive in the Client Master and a file is received against that client, the SPOC is notified immediately with the client details and source file reference.
Zvolv · Zai
Step 06 · Zai · Template Generation
Auto-Generate Invoice Draft Using Template Generation Rule
Once the billing type is confirmed, Zai looks up the Template Generation Rule Master (maintained in Zvolv) to retrieve the specific rule for that billing process type. A draft invoice file is created, with all available values populated by Zai.
The Template Generation Rule Master holds: Transaction Type → Template Headers → Value Calculation Logic → Default Values → Manually-Editable Fields → File Template. This master drives all template creation — future changes are managed here with no major rework.
The same standard template structure is used across all billing types — values are calculated and derived based on the specific transaction type being processed.
Zvolv · Zai
Zai assigns task → Invoice SPOC
Human Review Stage
Step 07 · SPOC · Review & Edit
Invoice SPOC Reviews, Edits & Re-uploads Draft
The Invoice SPOC receives the auto-generated draft via a task assignment. The SPOC reviews the file, makes any necessary modifications to the editable fields, and re-uploads the finalised draft back into Zvolv.
Zai pre-fills all available fields per the Template Generation Rule. SPOC only needs to verify and fill in fields marked as manually-editable in the rule master — significantly reducing processing time.
Randstad SPOC
Step 08 · Manager · Approval
Manager Review & Approval
The SPOC-reviewed file is automatically routed to the Manager for final approval via Zvolv's workflow. Once approved, the file is queued for SAP submission via API.
Rejected drafts are returned to the SPOC with comments via the Zvolv task system. Approval status is tracked in the audit trail.
Randstad Manager
Zvolv triggers SAP API integration automatically
System Integration
Step 09 · Zvolv · SAP API Integration
Zvolv Pushes Approved File to SAP via API
Upon Manager approval, Zvolv automatically submits the processed invoice file to SAP through the configured API integration — no manual data entry required. SAP processes the transaction and returns the response.
Automated API Push
Zero Manual SAP Entry
Real-time Response Capture
Step 10 · Zvolv · Post-Processing
SAP Response Captured → Open SO Master Updated
Zvolv collects the SAP response and maps the SO Number and Invoice Number generated for each entry against the uploaded file. These values are written back to the Open SO Master automatically.
SO Number → Open SO Master

Exception Handling
Exception Cases — Escalation & Audit Framework
All exceptions are auto-assigned to Invoice SPOC via Zvolv tasks and fully tracked for audit reporting.
1
No Key Values ExtractableIf Zai cannot fetch any key value data from the source file, the transaction is pushed to Invoice SPOC and tracked with reason logging.
2
Client Not in MasterIf the client details are not available in the Client Master, the transaction is handled manually by Invoice SPOC. Reason is captured and audited.
3
Client Marked InactiveIf a file is received for a client marked Inactive in the Client Master, the SPOC is notified immediately with full details.
Audit Dashboard: Zvolv tracks all exception assignments — number of billing transactions escalated to SPOC, exception type, frequency, and resolution status — available as a real-time audit report.
